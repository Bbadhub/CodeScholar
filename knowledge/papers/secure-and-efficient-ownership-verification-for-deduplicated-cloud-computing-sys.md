## Summary

The paper presents a secure and efficient proof of ownership (POW) scheme for deduplicated cloud computing systems, ensuring that only users with the complete file can prove ownership while minimizing I/O, communication, and computational overhead. Engineers should care because this approach addresses significant security vulnerabilities in existing deduplication systems.

## Key Contribution

**A secure and efficient ownership verification scheme that requires the user to possess the complete file to generate ownership proof while minimizing overheads.**

## Problem

The need to securely verify file ownership in deduplicated cloud systems to prevent unauthorized access by malicious users.

## Method

**Approach:** The method involves the server generating a random seed and challenging the user to prove ownership using the entire file and the random values. The user computes a response based on the file blocks and sends it to the server for verification.

**Algorithm:**

1. User sends file tag to server.
2. If file tag exists, server generates a random seed.
3. User computes response using the entire file and the seed.
4. User sends the response to the server.
5. Server computes its own response using the stored file and compares it with the user's response.
6. If responses match, access is granted.

**Input:** File tag (cryptographic fingerprint of the file), complete file.

**Output:** Proof of ownership response (hash value).

**Key Parameters:**

- `seed: random value generated by the server`
- `h: number of bits for the seed`
- `File blocks: segments of the file used in the computation`

**Complexity:** O(n) time for response generation, O(n) space for storing file blocks.

## Implementation Guide

**Data Structures:** File blocks, Hash tables for storing file tags

**Dependencies:** Cryptographic libraries for hash functions, Pseudorandom number generators

**Core Operation:**

```python
response = Hash(previous_response || Hash(file_block[i] || PRG(seed || i)))
```

**Watch Out For:**

- Ensure the hash function used is secure and collision-resistant.
- Properly manage the random seed to prevent replay attacks.
- Monitor for potential performance bottlenecks during file block processing.

## Abstract

When a new file is ingested, the server recognizes its digital fingerprint and says "I already have that one", no need to store another. Your storage quota still gets used, of course, and you still pay for the storage. But behind the scenes, there's only one actual copy of the file, with multiple pointers to it. You can see why this is great for cloud providers. Storage costs stay as low as possible, and bandwidth is lower than it would otherwise be, backup times shrink, and best of all: they get to charge multiple users for something they only actually did for one of them.
